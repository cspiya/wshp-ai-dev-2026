G4 · C5 checkpoint · 50 perc

Szándékból mechanikus garancia

Tanítási eredmény: el tudod dönteni, hogy egy elvárás egyszeri prompt, tartós szabály, újrafuttatható skill, eseményhez kötött hook vagy kötelező gépi kapu legyen; a szükséges tudást célzottan adod át, a védelmet pedig negatív próbával igazolod.

C5 · A rendszer új képessége: a repo megőrzi a tartós döntéseket, és a géppel eldönthető feltételeket automatikusan ellenőrzi. Az agent nem tud puszta narrációval „kész” állapotot jelenteni.
Erre építünk: a C3 specje megmondja, mit kell elérni; a C4-es RUG-kör függetlenül keresi a hibát. Most a visszatérő findingokból tartós tudást és regressziós védelmet készítünk.

1. Öt eszköz, öt felelősség

Ne dokumentummal próbáljunk tesztet helyettesíteni, és ne automatizáljunk olyan döntést, amelyhez üzleti ítélőképesség kell. .NET-ben sem ugyanaz a PR-leírás, a coding guideline, egy dotnet tool, a Git hook és a CI-ban futó architektúrateszt.

A hook az indítási mechanizmus; a gate az ellenőrzés és annak blokkoló eredménye. Egy hook futtathat gyenge parancsot, ezért a kettőt külön kell tesztelni.

2. Döntési fa: hova kerüljön egy tanulság?

Döntési doboz · Mi legyen a review-findingból?

Az alapprobléma: a chatben maradó tanulság elveszik; a túl hosszú szabálykönyvet viszont sem ember, sem agent nem priorizálja megbízhatóan.

A választásunk: előbb osztályozunk. Ha géppel eldönthető, gate; ha eseményhez kötött, hook indítja; ha ismételhető folyamat, skill; ha kontextuális elv, rövid rule; ha csak erre a feladatra igaz, prompt.

Miért ez? a legerősebb, még indokolt garanciát választjuk. Rövidebb marad az instrukció, csökken a token- és review-terhelés, és a determinisztikus hibát nem bízzuk modellmemóriára.

Alternatívák: mindent az AGENTS.md-be írni olcsó, de nem blokkol; mindent hookká tenni törékeny automatizmust ad; mindenre külön skillt írni túlzott karbantartás.

  1. Csak erre az egy feladatra igaz? Igen → prompt vagy spec-részlet.
  2. Több lépésben, ismételhetően végrehajtandó? Igen → skill, rögzített input/output szerződéssel.
  3. Egy eseménykor automatikusan induljon? Igen → hook vagy CI-trigger.
  4. A helyesség determinisztikusan mérhető? Igen → blokkoló gate és negatív fixture.
  5. Ítélőképességet igénylő tartós elv? Igen → rövid rule és független review.

3. Projektmemória és retrieval: mit kapjon meg az agent?

A projektmemória verziózott, kanonikus tudás: szabály, ADR, standard, spec és run log. A retrieval ennek célzott visszakeresése. .NET-analógia: nem az egész solutiont és wikit másoljuk egy ticketbe; Find All References, a legközelebbi projektfájl és a jóváhagyott ADR alapján állítjuk össze a munkacsomagot.

Tartós memóriaFeladatspecifikus context packetNe ide kerüljön
Mission, scope, AGENTS.md, kanonikus standard, ADR, valódi gate-parancs.Issue/spec, acceptance criteria, legközelebbi szabály, érintett contract és teszt.Teljes chat-történet, teljes repo-dump, duplikált standard, titok vagy elavuló deployment-állapot.

Döntési doboz · Repo-memória, nem modellmemória

Az alapprobléma: a modell belső tudása és a chat idővel elavul; a teljes kontextus pedig zajos és drága.

A választásunk: tartós döntés a repóba, aktuális külső állapot toolból, feladathoz csak a minimális packet.

Miért ez? visszakereshető a tulajdonos és a változástörténet; maker és reviewer ugyanazt a kanonikus forrást olvassa.

Alternatívák: RAG hasznos nagy dokumentációhoz, de kódhoz gyakran pontosabb az LSP és a strukturális keresés; chat-összefoglaló gyors, de nem source of truth.

  1. Adj át issue/specet és jóváhagyott acceptance criteriát.
  2. Linkeld a legközelebbi AGENTS.md-t és az egyetlen kanonikus standardot; ne másold őket promptonként.
  3. Válaszd ki csak az érintett slice-ot, contractot és tesztet.
  4. Add meg a valódi gate-parancsokat és az elvárt evidence-formátumot.
  5. Friss reviewer új packetet kapjon, a maker indoklása és ítélete nélkül.

4. BA–fejlesztő konvergencia: szándékból ellenőrzés

5. Gyakorlat: egy findingból blokkoló védelem

Cél: a toolkit public-content guardját helyes és hibás, kitalált mintán kipróbálni, majd a futási evidence-et rögzíteni.

  1. Olvasd el a toolkit/hooks/README.md fájlt, és azonosítsd az exit code-ok jelentését.
  2. Hozz létre egy ideiglenes, nem commitolandó invented-safe.md fájlt ezzel: Fiktív workshop-minta, publikus használatra.
  3. Futtasd: 
    node toolkit/hooks/guard-public-content.mjs invented-safe.md
    Várt eredmény: exit code 0.
  4. Hozd létre az ideiglenes negatív fixture-t úgy, hogy maga a publikus notebook ne tartalmazza egyben a tiltott markert: 
    Set-Content invented-blocked.md ("TODO" + "-PRIVATE remove before publish.")
  5. Futtasd: 
    node toolkit/hooks/guard-public-content.mjs invented-blocked.md
    Várt eredmény: exit code 1 és a blokkolás oka a stderr-ben.
  6. Futtasd a regressziós suite-ot: 
    node --test toolkit/hooks/hooks.test.mjs
    Várt eredmény: a safe út, a blokkoló út, a child failure és a timeout tesztje zöld.
  7. Töröld a két ideiglenes fájlt. Az issue/run logba írd a három parancsot, exit code-ot és a releváns összegzést; környezeti titkot ne másolj.
Kész, ha: a helyes minta átmegy, a tiltott marker elbukik, a regressziós suite zöld, és másik résztvevő az evidence-ből meg tudja mondani, mi futott és mit bizonyított.

Tipikus elakadások: rossz working directory; hiányzó Node.js; csak a „nem működik” állítás kerül a logba; a tesztmarker bent marad a repóban.

Plan B: ha a lokális Node nem működik, a tréner futtatja ugyanezeket a parancsokat a tiszta referencia-repóban, a résztvevő pedig a parancs–exit code–állítás hármast elemzi. CI-only fallback: node toolkit/hooks/run-stop-checks.mjs toolkit/hooks/checks.smoke.json.

6. Alkalmazási bizonyíték: a referencia-app csak próbaterhelés

Nem új feature-t építünk. A referenciaalkalmazás meglévő modulhatára azt teszteli, hogy az operating model meg tud-e állítani egy szabályszegést.

Validációs csomag:
  1. cd reference-app
  2. npm run lint — a boundary szabályok aktuális, zöld állapota.
  3. npm run test -- src/platform/lint-boundaries.test.ts — valódi tiltott import-fixture bukik, a megengedett út átmegy.
  4. A run log hivatkozza a specet, parancsot, eredményt és korlátot: a lint architektúra-invariánst igazol, üzleti működést nem.

Modell- és harnessfüggetlen mag: a szabály szándéka, a context packet, a futtatott parancs, az exit code és a negatív fixture változatlan. Csak a skill helye és a hook eseményének konfigurációja adapterfüggő.

7. Vidd haza

A prompt kér, a szabály vezet, a skill ismétel, a hook elindít, a gate megállít — a repo-memória pedig a következő agentnek is átadja, miért.
