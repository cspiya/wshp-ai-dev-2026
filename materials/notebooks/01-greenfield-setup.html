1. Cél és kész állapot

Miért ezzel kezdünk? A coding agent csak akkor tud megbízhatóan dolgozni, ha a repónak ismert küldetése, határa, közös szakmai léce, valódi belépési pontja és futtatható ellenőrzése van. A helyi „nálam működik” még nem szállítási bizonyíték.

Döntési doboz · A szállítópálya bizonyíték, nem a termék

Az alapprobléma: az infrastruktúra önmagában nem mondja meg az agentnek, mit szabad építenie és mit jelent a jó munka.

A választásunk: a starterben előbb lezárjuk a C1–C2 bootstrapot, majd ugyanannak a repónak a buildjével és preview-jával igazoljuk, hogy a szerződés végrehajtható.

Miért ez? Így a CI nem díszlet: a kanonikus Definition of Done gépi része.

Alternatívák: előbb deployolni gyorsabb, de csak a technikai hordozót bizonyítja; az összes szabályt egy promptba írni nem ad verziózott, csapatban review-zható szerződést.

A gyakorlat akkor kész, ha mind a nyolc bizonyíték megvan:
  • az AGENTS.md megnevezi a küldetést, scope-ot, tiltásokat és valódi done-parancsokat;
  • maker és reviewer ugyanarra a kanonikus standardra és Definition of Done-ra hivatkozik;
  • egy read-only agent helyesen mondja vissza a repo-szerződést;
  • a projekt lokálisan elindul;
  • a typecheck, lint, teszt és build zöld;
  • a saját GitHub-repóban a CI zöld;
  • a main branchnek van működő production URL-je;
  • egy külön pull requesthez saját Vercel preview URL tartozik; a Neon-blokk után ehhez külön DB-branch is létrejön.
Időkeret: egy külső fiók- vagy OAuth-hibával legfeljebb 10 percet tölts. Utána válts a Plan B-re, és haladj tovább.

4. Agent-ready bootstrap — a repo működési szerződése

Cél: az agent találgatás nélkül meg tudja mondani, mi ez a repo, meddig terjed a feladata, és mivel kell bizonyítania a kész állapotot. .NET-es analógia: a solution README-je, coding standardja, architecture testje és pipeline Definition of Done-ja együtt — nem egy chatben elvesző instrukció.

4.1Írd le a repo identitását

  1. Nyisd meg az AGENTS.md fájlt a participant-repo/ gyökerében.
  2. Angolul írj a technikai stack elé egy rövid Mission részt: a repo agent-ready működést épít, az alkalmazás annak validációs workloadja.
  3. Angolul adj In scope és Out of scope listát. A workshopon csak kitalált mintaadat engedélyezett; ügyféladat, valódi ár, személyes adat és titok tiltott.
  4. Ellenőrizd, hogy a fájl pontosan a package.json-ban létező parancsokat nevezi meg.

4.2Nevezd meg az egyetlen minőségi lécet

  1. Hozz létre egy docs/engineering-standard.md fájlt a participant-repo/ alatt.
  2. Angolul rögzítsd benne röviden a típusság, modulhatár, tesztelés, biztonság és publikus adatkezelés elvárását.
  3. Az AGENTS.md hivatkozzon erre a fájlra mint kanonikus standardra; ne másold ugyanazt a szabálylistát több helyre.
  4. Angolul írd le a Definition of Done-t: scope betartva, acceptance criteria bizonyítva, diff review-zva, typecheck/lint/test/build zöld, titok nem került commitba.

Read-only szerződésellenőrző prompt

Read AGENTS.md, package.json, docs/engineering-standard.md, and the CI workflow.
Do not modify files. Report:
1. the repository mission,
2. in-scope and forbidden work,
3. the single canonical engineering standard,
4. the exact commands that define done,
5. any contradiction or missing decision as DECISION REQUIRED.

4.3BA/dev konvergencia

  1. A BA-szerep ellenőrizze: a mission és scope üzletileg érthető-e, van-e kitalálandó szabály.
  2. A dev-szerep ellenőrizze: minden done-parancs tényleg létezik-e, a standard hivatkozása egyértelmű-e.
  3. Hasonlítsátok össze a két listát az agent visszamondásával.
  4. Az eltéréseket a fájlokban javítsátok; hiányzó üzleti döntésnél írjatok DECISION REQUIRED pontot.
Bootstrap kész, ha:
  • a read-only agent nem talál ellentmondást;
  • a BA és a fejlesztő ugyanazt érti a repo célján és a done állapoton;
  • minden felsorolt parancs megtalálható a package.json-ban;
  • a standard egyetlen kanonikus fájl, nem másolt szabályhalmaz.
Tipikus elakadások és Plan B
  • Túl általános mission: nevezd meg külön az operating modelt és a validációs workloadot.
  • Ellentmondó parancsok: a package.json a futtatható forrás; az instrukciót igazítsd hozzá.
  • Az agent írni akar: utasítsd újra read-only módra, vagy Plan B-ként a társad mondja vissza a contractot.
  • Nincs idő teljes standardra: használd a tréner által adott minimális mintát, jelöld a bővítendő döntéseket, de ne hagyj üres placeholdert.

5. Determinisztikus telepítés és lokális kapuk

A workshop-lab/participant-repo/ gyökerében, a committed lockfile alapján telepíts, majd futtasd az összes kaput:

npm ci
npm run typecheck
npm run lint
npm run test
npm run build

Indítsd el az alkalmazást:

npm run dev

Nyisd meg: http://localhost:3000. Ha betöltött, állítsd le Ctrl+C-vel.

Lockfile-szabály: ha dependency-t adsz hozzá, npm 10-zel frissítsd: npx npm@10 install <csomag>. A CI Node 22 + npm 10 környezetet használ.

Minden kapuról írj egy sort a workshop-lab/workshop-evidence/g1-bootstrap.md privát fájlba: parancs, exit code, egymondatos eredmény-összefoglaló. Commit után egészítsd ki a git rev-parse HEAD teljes SHA-jával. A nyers logot vagy titkot ne másold bele.

Lokális done-kritérium
  • Mind a négy kapu 0 exit code-dal zárult.
  • A kezdőoldal betöltődött.
  • A bizonyítékfájlban mind a négy parancs, exit code és összefoglaló szerepel.
  • Nincs commitra váró .env.

6. Saját GitHub-repo és az első zöld CI

Publikus repo-higiénia: csak kitalált mintaadatot használj. Ne másolj be ügyfélkódot, belső dokumentumot, valódi árat, személyes adatot, API-kulcsot vagy connection stringet. Ha az ötleted bizalmas, a workshopra készíts semleges, kitalált változatot.

6.1Üres repó létrehozása

  1. GitHub → New repository.
  2. A név egyezzen a lokális projekt nevével.
  3. A workshopon válaszd a Public láthatóságot.
  4. Ne kérj README-t, .gitignore-t vagy license-t.
  5. Kattints a Create repository gombra.

6.2Első commit és push

A nagybetűs helyőrzőket cseréld a saját értékeidre:

git add .
git commit -m "Bootstrap workshop project"
git rev-parse HEAD
git remote add origin https://github.com/SAJAT-GITHUB-NEVED/SAJAT-REPO-NEVED.git
git push -u origin main

A kiírt teljes commit SHA-t másold a workshop-lab/workshop-evidence/g1-bootstrap.md fájlba a négy kapu mellé.

GitHub → Actions → CI. Várd meg, amíg zöld.

Ha nincs CI-futás: ellenőrizd, hogy a saját repóban létezik-e a .github/workflows/ci.yml. Ha hiányzik, a starter másolása nem volt teljes.

7. Első production deployment Vercelen

Feltételes UI-leírás, ellenőrizve: 2026-07-12. A Vercel feliratok vagy menühelyek változhatnak. Ha az alábbi útvonal nem egyezik, keresd az új projekt létrehozását és a Git repository importját; Plan B-ként a tréner végigmutatja a saját projekten, te pedig a lokális és GitHub CI-evidence-et zárod le.
  1. Nyisd meg a Vercel New Project oldalt.
  2. A GitHub-repód mellett kattints az Import gombra.
  3. Ha nem látod, a GitHub-integrációnál adj hozzáférést ehhez a repóhoz.
  4. Framework Preset: Next.js.
  5. Root Directory: ./. A saját repódban az app a gyökérben van.
  6. A build és install beállításokat hagyd automatikus felismerésen.
  7. Environment variable még nem kell.
  8. Kattints a Deploy gombra.

Nyisd meg a generált URL-t. A kezdőoldal töltsön be, a deployment legyen Ready, és a main aktuális commitjára mutasson.

Első szállítási bizonyíték: másold ki a production URL-t a jegyzeteidbe. Ez nem ugyanaz lesz, mint a PR-preview linkje.

9. Neon bekötése és DB-branch preview-nként

Miért? Külön web-preview közös adatbázissal nem valódi izoláció. Egy hibás migráció vagy tesztadat minden más preview-t is érintene.

Ezt akkor végezd el, amikor a tréner jelzi. A starter kezdetben szándékosan nem használ adatbázist. A branchelés infrastruktúráját most beköthetjük, az alkalmazásos izolációs próbát az első DB-backed feature után zárjuk.
Feltételes UI-leírás, ellenőrizve: 2026-07-12. A Neon/Vercel Marketplace feliratai és az automatikus preview-branch opció elhelyezése változhat. Ne találgass: használd az aktuális dokumentációt vagy a tréner előre provisionált projektjét.

9.1Disposable Neon projekt létrehozása és kiválasztása

  1. A Neon Console-ban hozz létre egy új, eldobható workshop-projektet, például participant-workshop-lab néven, vagy kérd el a tréner által előre provisionált Plan B projekt pontos nevét.
  2. Jegyezd fel a projekt nevét és azonosítóját a privát g1-bootstrap.md fájlba; connection stringet és jelszót ne rögzíts.
  3. Nyisd meg a Vercel projektedet, és keresd a Marketplace/Storage integrációk között a Neont.
  4. Kapcsold össze a saját Neon-fiókodat, vagy Plan B-ként válaszd a tréner által megnevezett, előre provisionált integrációt.
  5. Az integrációban explicit módon ezt az eldobható projektet és ezt az egy Vercel projektet válaszd ki.
  6. Ha a felület kínál preview deploymentenkénti DB-branch opciót, kapcsold be; ha nem, a Neon Console-ban hozz létre kézzel egy preview-chore-verify-preview branchet, és a tréner segítségével rendeld a Preview környezethez.
  7. Ha elérhető, kapcsold be az obsolete preview branchek automatikus törlését; különben a gyakorlat végén kézzel töröld.

9.2Environment variable ellenőrzése

  1. Vercel → Project → Settings → Environment Variables.
  2. Ellenőrizd, hogy létezik a DATABASE_URL.
  3. Legalább a Production és Preview környezetben legyen elérhető.
  4. Az értéket ne másold chatbe, issue-ba, képernyőképbe vagy commitba.

9.3Lokális kapcsolat

Neon Console → Connect: másold ki a development/production branch connection stringjét. A projekt gyökerében:

Windows PowerShell
Copy-Item .env.example .env
macOS / Linux
cp .env.example .env

Az .env fájlba kerüljön:

DATABASE_URL=postgresql://...

Ellenőrizd, hogy Git ignorálja:

git status --short
git check-ignore .env

A második parancsnak ki kell írnia: .env.

9.4A preview DB-branch bizonyítása

  1. Ha a PR az integráció előtt készült, pusholj egy új commitot, vagy indíts redeployt.
  2. Várd meg a Vercel preview Ready állapotát.
  3. Nyisd meg a Neon Console integrációs/Branches nézetét.
  4. Keresd meg a PR-hez vagy a chore/verify-preview branchhez tartozó DB-branchet.
  5. Jegyezd fel a branch nevét, de connection stringet vagy jelszót ne rögzíts.
Az első DB-backed feature után futtatandó izolációs próba
  1. A preview URL-en hozz létre egy egyedi tesztadatot.
  2. Ellenőrizd, hogy productionben nem jelenik meg.
  3. Zárd le a PR-t vagy töröld a feature branchet.
  4. Ellenőrizd, hogy a preview DB-branch obsolete lett vagy törlődött.

Checkpoint elfogadva, ha: az agent helyesen mondja vissza a missiont, scope-ot, kanonikus standardot, valódi parancsokat és határokat, te pedig külön meg tudod mutatni a lokális, CI-, preview- és DB-branch bizonyítékot.
Vidd haza: az agent-ready repo nem egy telepített coding agent. Verziózott működési szerződés, közös minőségi léc és olyan szállítópálya, amely megfigyelhető eredménnyel bizonyítja a szabályok végrehajtását.

Mi marad stabil másik modell vagy agent harness mellett? A mission, scope, standard, Definition of Done, CI-parancsok és evidence-léc változatlan. Csak a végrehajtó indítása és eszközkapcsolata cserélődik; ugyanazzal a read-only contract prompttal újra ellenőrizhető.
