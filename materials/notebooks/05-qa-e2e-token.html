G5 · C6 checkpoint · 55 perc

QA, E2E és context budget

Tanítási eredmény: egy jóváhagyott üzleti változtatáshoz kockázatalapú bizonyítékláncot tervezel, külön ellenőrzöd a fake és a valódi adapter szerződését, majd egy hitelesített preview ellen futó böngészős teszttel validálod a teljes agent-ready működést.

C6 · A rendszer új képessége: a repo nemcsak kódot állít elő, hanem a spectől a valódi futási környezetig visszakövethető, egymásra épülő bizonyítékot szolgáltat.
Erre építünk: a C3 acceptance criteria megmondja, mit jelent a kész; a C4 RUG függetlenül támadja az eredményt; a C5 gate-jei mechanikusan blokkolnak. Most reprezentatív alkalmazási terheléssel ellenőrizzük, hogy a teljes rendszer együtt működik-e.

1. Az alkalmazás szerepe: validation workload

A referenciaalkalmazás nem a workshop végterméke. Validation workload: üzleti szabályt, modulhatárt, adatot, változó külső függőséget, preview deploymentet és böngészős utat ad az operating model acceptance tesztjéhez.

.NET-analógia: nem azért építünk egy teljes webes slice-ot, hogy bemutassuk az ASP.NET controllert, hanem hogy együtt terheljük a domain-szabályt, az EF-adaptert, a DI-drótozást, a pipeline-t és a Playwright/Selenium kritikus utat.

Döntési doboz · Kockázathoz bizonyítékréteg

Az alapprobléma: egy zöld unit teszt nem bizonyítja a drótozást; egy kézzel kattintott preview nem bizonyítja a határesetet; egy nagy E2E suite lassú és nehezen diagnosztizálható.

A választásunk: minden kockázathoz a legolcsóbb hiteles tesztréteget rendeljük, majd egy keskeny E2E úttal igazoljuk az integrációt.

Miért ez? gyors visszajelzést és pontos hibaforrást kapunk, miközben a valódi wiring sem marad ellenőrizetlen.

Alternatívák: „mindent E2E-ben” lassú és törékeny; „mindent mockkal” hamis rendszert igazolhat; kizárólag manuális QA nem reprodukálható és nem regressziós védelem.

4. Gyakorlat A: kockázatból evidence-terv

Cél: bizonyítékláncot készíteni a fiktív workshop-regisztráció kritikus útjához anélkül, hogy egyetlen óriás E2E tesztbe zsúfolnánk mindent.

  1. Olvasd el a jóváhagyott acceptance criteriát és a reference-app/src/modules/registrations/, pricing/, checkout/ contractjait. A slice-on kívüli belső fájlokat csak indokoltan add a kontextushoz.
  2. Írj fel négy kockázatot: árkerekítés, fizetési elutasítás, státusz-perzisztencia, böngészős drótozás.
  3. Mindegyikhez rendeld a legolcsóbb hiteles réteget: domain unit; port/alkalmazási teszt; repository contract; preview E2E.
  4. Minden sorhoz írd oda: mit bizonyít és mit nem bizonyít.
  5. A BA ellenőrizze, hogy minden acceptance criterion legalább egy bizonyítékhoz kötődik. A fejlesztő ellenőrizze, hogy nincs két drága réteg ugyanarra az állításra indok nélkül.
  6. A reviewer kapja meg a specet, a kockázat–teszt mátrixot és a kanonikus standardot, de ne kapja meg a maker „minden rendben” ítéletét.
Kész, ha: minden jóváhagyott viselkedéshez van megfelelő bizonyítékréteg, minden réteg korlátja le van írva, és az E2E csak az integrációs kritikus utat viszi.

Tipikus elakadások: technológiai lista készül kockázati térkép helyett; a mockot valódi adapter-evidence-ként kezeljük; a BA csak az elkészült UI-t látja, a hibaágat nem.

Plan B: ha a referencia-app nem indul, készítsd el a mátrixot a modul contractokból és a meglévő tesztnevekből. A futási cellákat jelöld nem futott állapotúra; a tervezés elkészülhet, a validációs gate nem.

5. Gyakorlat B: contracttól a preview E2E-ig

Cél: ugyanahhoz a commithoz kötött lokális, adapter- és böngészős bizonyítékot rögzíteni. A referencia-app itt a keretrendszer próbaterhelése, nem fejlesztési feladat.

  1. Rögzítsd a vizsgált commit SHA-ját: 
    git rev-parse HEAD
    Várt eredmény: egyetlen teljes commit-azonosító.
  2. A reference-app mappában futtasd külön, hogy az exit code rétegenként látható legyen: 
    npm run typecheck
npm run lint
npm run test
    Várt eredmény: mindhárom exit code 0. Jegyezd fel a tesztösszegzést és minden skipet.
  3. Ellenőrizd a repository contract suite-ot. Ha van eldobható Neon branchhez tartozó TEST_DATABASE_URL, futtasd a suite-ot azzal is. Várt eredmény: ugyanaz a viselkedési suite zöld az in-memory és Drizzle adapteren, releváns skip nélkül.
  4. Szerezd meg a commit/PR hitelesített preview deployment URL-jét GitHub/Vercel állapotból. Ellenőrizd a projektet, branchet, commitot és a sikeres deployment státuszt; URL-t ne találj ki és ne másolj titkot.
  5. PowerShellben futtasd a kritikus böngészős utat: 
    $env:PLAYWRIGHT_BASE_URL="https://<verified-preview-url>"
npx playwright test --grep @happy-path
    Várt eredmény: a happy path zöld a hitelesített preview ellen.
  6. Kérj friss kontextusú review-t a specre, diffre, pontos parancsokra és eredményekre együtt. Minden findingot reprodukálj, mielőtt javítanád.
  7. Rögzítsd az evidence-csomagot: commit; parancs; local/preview/isolated DB környezet; exit code; rövid eredmény; skip; mit bizonyít; mit nem; maradék kockázat.
Kész, ha: másik ember reprodukálni tudja, mely adaptereken futott a contract, mely commit mely preview-ját tesztelted, és mi maradt bizonyítatlan.

Tipikus elakadások: a lokális in-memory E2E-t preview evidence-ként jelentjük; a DB-suite skipje elvész; régi deployment URL fut; a „green” állítás mellől hiányzik a parancs vagy SHA.

Plan B: ha nincs preview/auth/hálózat, futtasd a lokális npm run test:e2e in-memory seamet. Jelöld: lokális wiring evidence, nem preview- vagy valódi DB-bizonyíték. A tréner felvétele vagy előre rögzített zöld run logja bemutathatja a lépést, de a résztvevő saját preview gate-je nyitva marad.

6. Context budget és token: a legolcsóbb megbízható visszacsatolás

A cél nem a legkevesebb agenthívás, hanem a teljes, elfogadott eredmény legalacsonyabb költsége. A túl sok kontextus elrejti a releváns szabályt; a túl kevés találgatásra kényszerít.

Döntési doboz · Keskeny packet, fokozatos bizonyítás

Az alapprobléma: teljes repo + teljes chat drága és zajos; a korai E2E lassú; a bizonyítékok elhagyása viszont visszanyitást okoz.

A választásunk: először spec, helyi szabály, érintett contract és gyors gate; csak a zöld diff kerül független review-ra és preview E2E-re.

Miért ez? a gyors determinisztikus kapu olcsón szűkít, a drágább agent- és böngészőidő pedig már kisebb, releváns felületet vizsgál.

Alternatívák: egyetlen hatalmas sessionben nő a stale assumption; minden fájl összefoglalása duplikálja a forrást; review és E2E kihagyása olcsónak látszik, de a hibát későbbre tolja.

7. Evidence-átadási szerződés

A build log és képernyőkép önmagában nem handoff. A kanonikus állapot az issue/spec és a Git commit; a rövid trace ezekhez köti a futást.

Evidence item
- Source: issue / commit / verified preview
- Command: exact command, secrets redacted
- Environment: local | preview | isolated DB branch
- Result: exit code + concise test summary
- Proves: one bounded claim
- Does not prove: explicit limitation
- Remaining risk / skipped gate: none | description

BA ellenőrzése: a „Proves” sor megfelel az acceptance criteriumnak. Fejlesztő ellenőrzése: a parancs és környezet alkalmas erre az állításra. Csak együtt adnak konvergenciát.

8. Modell- és eszközcserélhetőség

Változatlan marad a spec, a kockázat–teszt mátrix, a contract suite, a gate-parancs, az evidence-formátum és a preview acceptance. Csak az agent indítása, toolneve és auth-adaptere változik. Az új végrehajtó akkor elfogadható, ha ugyanazon reprezentatív workloadon ugyanazt a minőségi lécet teljesíti — nem attól, hogy újabb vagy magabiztosabb választ ír.

9. Vidd haza

Az állítás nem evidence: a kockázathoz illő teszt, a megnevezett környezet, az exit code és a reprodukálható kritikus út együtt bizonyít.
