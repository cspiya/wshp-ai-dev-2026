## 2. Money math pinned to the accepted mock

Domain (`src/modules/orders/domain/order.ts`, integer HUF only):

- net subtotal = Σ unitNet × qty; qty 1..5, above 5 the message names the
  custom group offer.
- coupon (invented catalog, `WELCOME10` = 10% off net) **floors**.
- VAT 27% of net-after-discount rounds **half-up** via
  `floor((net × 27 + 50) / 100)` — no floating point.
- Worked numbers reproduced exactly in `order.domain.test.ts`:
  38 500 → −3 850 → 34 650 → VAT 9 356 (9 355.5 half-up) → gross 44 006;
  plus the .5 boundary (50 → 13.5 → 14) and the discount floor (105 → 10).
- Per-seat gross helper `grossFromNet` matches the mock card prices
  (10 000 → 12 700, 18 500 → 23 495, 14 000 → 17 780).

## 3. Decisions (with reasons)

1. **Contact carries `name`** (spec bullet said `{email, phone}`): the
   frontend spec requires name/email/phone as required contact fields and
   the accepted mock's confirmation ledger prints
   `Contact: Kiss Anna · phone · email` — impossible without a stored
   contact name. Deviation said out loud here.
2. **Person-mode buyer name = the contact full name** (mock-faithful): the
   mock's billing module shows extra fields only in company mode; person
   mode shows a mini note ("the buyer name is the full name from the
   contact block") instead of a duplicate input. A `buyer.name` validation
   error is surfaced on the Full name field.
3. **Shop UI lives in `src/modules/orders/ui/`**, not `src/app/shop/`
   (golden-path form, same as `workshops/ui/`). First attempt kept the
   views in `src/app/shop` importing `orders.contract.ts` — that dragged
   the router factory and `@trpc/server` into the client bundle and crashed
   `/shop` at hydration ("You're trying to use @trpc/server in a non-server
   environment"). Client components now import the domain RELATIVELY; the
   server page imports `ShopJourney` through the contract. The contract
   documents the rule.
4. **Unknown coupon is rejected in the router** (`BAD_REQUEST`,
   `Unknown coupon "X" — remove it or use a valid code.`) using the domain's
   `resolveCoupon`/`unknownCouponMessage`, not a zod refine — so the client
   shows a clean human message instead of a serialized issue list.
5. **The server sources prices from the catalog — clients cannot influence
   price or title.** (Corrected after review: the first cut accepted
   `unitNetHuf`/`title` from the client, so a guest could have bought at
   1 HUF.) The item input schema is `{ workshopId, quantity }` only (max 20
   lines, duplicate workshop ids rejected); `preview` and `place` resolve
   every line through an injected workshop source (composition root adapts
   the workshops repo — same pattern as the registrations schedule) and run
   the money math on the catalog's title + list price. A smuggled `totals`,
   `unitNetHuf` or `title` key is stripped by the plain (non-strict) zod
   object; pinned by router tests including a price-tamper test.
6. **Superseded on 2026-07-14 (§10):** the initial implementation kept
   `checkout.authorize` protected, so the deployed “guest” journey silently
   required sign-up. That contradicted the accepted requirement even though
   the local in-memory seam hid it. The fake PaymentPort procedure is now
   public and schema-validated; unrelated user-owned mutations stay protected.
7. **"Download summary" from mock 5/5 was NOT built** — it had no real
   behavior to back it (no PDF/print pipeline), and a dead button would be a
   fabricated affordance.
8. **Billing postal code and city are separate fields** (mock combines them
   in one input) — the domain address has four required parts and separate
   fields validate honestly.
9. **Cart adds navigate to the cart view** (catalog "Add to cart" adds 1
   seat; product page adds the stepper quantity) so the mock's cart-count
   chip is always visibly truthful.

## 9. Limitations / follow-ups

- Drizzle order persistence deferred (§4) — orders are process-local.
- `orders.place` records the payment authorization id but cannot VERIFY it
  against the PaymentPort (the port has no lookup op and stays untouched) —
  a client could pass an arbitrary string. The compound risk, said plainly:
  an order can be placed with an arbitrary auth string, i.e. without any
  real authorization having happened; verification would need a PaymentPort
  lookup operation, which is a deliberate teaching boundary. Acceptable
  here; a real system would verify or capture server-side.
- **"Seats N" is catalog metadata, not enforced inventory**: the capacity
  shown on the catalog cards and the product page is display-only — orders
  do not decrement or validate seat capacity, so overselling is possible by
  design at this teaching boundary.
- **HUF amounts render with en-US grouping** (`10,000 HUF`) while the
  accepted mock used Hungarian space grouping (`10 000 HUF`) — a deliberate
  single-locale choice (`formatHufAmount` pins `en-US`, consistent with the
  app's English UI), not an oversight.
- Checkout form state (contact/billing/coupon) resets if the buyer
  navigates back to the cart — wizard memory beyond the cart itself was not
  in scope.
- Coupon totals during an applied-unknown-coupon error: the summary box
  intentionally goes "Totals unavailable — fix the coupon below." and the
  pay button disables until the coupon is removed.

## 10. 2026-07-14 remediation — guest means signed out

**Trigger:** the WEN-339 preview deployment exposed an inherited checkout
failure. Vercel deployed successfully, but `E2E on preview` timed out waiting
for `Sample Workshop: Checkout Flow`. The title existed only in the local
in-memory seam. The same spec also signed up on every preview because the fake
payment procedure was protected, contradicting “account optional.”

**Fix:** `checkout.authorize` is a public, schema-validated procedure because
this adapter moves no real money and the accepted webshop contract is public
guest checkout. The fake PaymentPort itself and all unrelated protected
procedures are unchanged. The non-operational account panel was removed from
the checkout UI; the page now states that authentication is a separate future
integration. The browser spec creates two uniquely named workshop fixtures
with the exact prices its money assertions require, buys them while signed out,
and deletes both after success. No seed title, DB snapshot, or auth runtime is
part of the test precondition.

**Evidence:** focused checkout/PaymentPort tests 7/7; typecheck and lint; full
unit/contract suite 100 passed with the expected 11 DB-gated skips; Next.js
production build; and both local Playwright happy paths pass. A fresh-context
RUG review found two documentation-truthfulness issues; both were fixed and
passed bounce-back review. PR #12's Vercel deployment succeeded and its
[signed-out Preview E2E run](https://github.com/cspiya/wshp-ai-dev-2026/actions/runs/29290588293)
passed in 59 seconds on 2026-07-14.
